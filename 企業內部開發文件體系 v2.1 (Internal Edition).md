# 🧬 企業內部開發文件體系 v2.1 (Internal Edition)

### 📏 專案分級 (T-shirt Sizing)

內部開發講求「快」與「穩」，不需過度文檔化，依照影響範圍分級：

* **S (個人/小組工具)**：僅需 **A (簡易版) + B**。例如：自動整理某個 Excel 的 Python Script。
* **M (部門級應用)**：**A + B + C**。例如：業務部的報價單生成系統（多人使用）。
* **L (跨部門/全公司系統)**：**全套 A - F**。例如：公司內部的請假系統或 ERP 外掛模組。

---

### 📚 文件架構總覽

| 文件 | 核心角色 | 內部開發觀點 (修改重點) | 形式建議 |
| --- | --- | --- | --- |
| **A. Project Charter** | 需求方 / 開發者 | 定義「省下多少工時」與「不做什麼」 | 1-Pager (Wiki/Doc) |
| **B. Functional Spec** | PM / 開發者 | 定義「作業流程」與「權限控管」 | Doc + Mockup |
| **C. Tech Design** | 開發者 | 定義「現有資源整合」與「選型理由」 | Doc + Diagram |
| **D. Interface Spec** | 開發者 | 定義「輸入/輸出格式」 (含檔案/DB) | Code / Schema |
| **E. Data Governance** | DB / 資安 | 定義「資料來源」與「機敏性」 | Doc / SQL |
| **F. Support & Ops** | 維運 / IT | 定義「維運邊界」與「故障排除」 | Wiki (必備) |

## A. Project Charter（專案憲章）

📌 **立案層**：保護開發者，明確定義這不是「許願池」。

| 區塊 | 內容與用途 | **💡 內部開發亮點** |
| --- | --- | --- |
| **Background & Pain** | 現狀與痛點 | 目前人工流程多痛苦？容易錯哪裡？(As-Is Process)。 |
| **Trigger** | **觸發原因** | **(替換 Why Now)** 為什麼現在要做？(長官要求 / 舊系統停用 / 法規變更)。 |
| **ROI / Efficiency** | **效益分析** | **(替換 Metrics)** 預計節省多少人時 (Man-hours)？降低多少錯誤率？ |
| **Scope In / Out** | **專案邊界** | **(最重要)** 絕對要寫「**不包含**什麼功能」。(拒絕無效的 Feature Creep)。 |
| **Requestor & Owner** | 需求與負責方 | 誰提的需求？(驗收找他) / 誰開發？(有問題找你)。 |
| **Timeline** | 預計時程 | 只有在需求凍結的前提下，承諾的上線日。 |

---

## B. Functional Spec（功能規格書）

📌 **需求層**：確保做出來的東西是同事真正要的流程。

| 區塊 | 內容與用途 | **💡 內部開發亮點** |
| --- | --- | --- |
| **Workflow Steps** | **作業流程** | **(替換 User Story)** Step 1-2-3，使用者如何操作此工具完成任務。 |
| **Access Control** | **權限矩陣** | **(新增)** 誰能看？誰能改？(例如：只有經理能看薪資欄位)。 |
| **Input Logic** | 輸入邏輯 | 防呆機制 (Validation)。內部工具最怕同事輸入髒資料搞掛系統。 |
| **Audit Logs** | **稽核紀錄** | **(替換 Telemetry)** 記錄「誰」在「何時」改了什麼資料？(責任釐清用)。 |
| **Dependency** | **外部依賴** | 依賴哪個部門的 Excel 或 API？若對方改格式，本系統不負責。 |
| **DoD** | 驗收標準 | Definition of Done (例如：產出的報表需與人工核算一致)。 |

---

## C. Tech Design（技術設計書）

📌 **實作層**：強調「低成本」與「可維護性」。

| 區塊 | 內容與用途 | **💡 內部開發亮點** |
| --- | --- | --- |
| **Context Diagram** | 系統關聯圖 | 這個工具在公司現有 IT 版圖中的位置 (連誰的 DB？發信給誰？)。 |
| **Build vs Buy** | **選型評估** | 為什麼要寫 Code？能不能用 Excel VBA / Power Automate / 現成 SaaS 解決？ |
| **Tech Stack** | 技術棧 | 盡量使用公司「現有」技術 (避免引入沒人會維護的新語言)。 |
| **ADR Log** | 決策日誌 | 記錄那些「因為時程趕，所以先 Hard-code」的技術債 (Tech Debt)。 |
| **Resource Plan** | **資源需求** | 需要申請 VM 嗎？需要開防火牆嗎？需要申請 DB 帳號嗎？ |

---

## D. Interface Spec（介面與整合書）

📌 **合約層**：內部系統常透過檔案或 DB 交換資料，需嚴格定義。

| 區塊 | 內容與用途 | **💡 內部開發亮點** |
| --- | --- | --- |
| **Input Format** | **輸入格式** | 若是匯入 Excel/CSV，**嚴格定義**欄位名稱、順序、格式 (Template)。 |
| **DB View / API** | 資料介面 | 若是讀取 ERP，是讀哪個 View？若是 API，Swagger 在哪？ |
| **Output Format** | 輸出定義 | 產出的報表長怎樣？寄出的 Email 內容模板。 |
| **Error Handling** | 錯誤回饋 | 當輸入資料錯誤時，如何報錯給使用者？(彈窗 / Log / Email)。 |

---

## E. Data Governance（資料治理書）

📌 **數據層**：內部資料最重「安全」與「來源」。

| 區塊 | 內容與用途 | **💡 內部開發亮點** |
| --- | --- | --- |
| **Data Source** | **資料來源** | 每一筆資料是從哪裡來的？(Truth of Source)。避免引用了錯誤的二手資料。 |
| **Schema / ERD** | 資料結構 | 涉及的 Table 結構。 |
| **Sensitivity** | **機敏分級** | 是否包含個資 (PII)、薪資、成本？需不需要遮罩或加密？ |
| **Data Lifecycle** | 資料週期 | 暫存檔多久刪除？Log 保留多久？ |

---

## F. Support & Ops（維運手冊）

📌 **生存層**：**一定要寫！** 避免離職或休假時電話響不停。

| 區塊 | 內容與用途 | **💡 內部開發亮點** |
| --- | --- | --- |
| **Support Window** | **維運時間** | **(重要)** 宣告「非上班時間不處理」或「僅提供 Best-effort 支援」。 |
| **Troubleshooting** | **故障排除** | 常見問題 Q&A (FAQ)。例如：「按鈕沒反應通常是...」、「報表空白是因為...」。 |
| **Escalation** | 後送流程 | 我處理不了要找誰？(例如：網路不通找 MIS，DB 連不上找 DBA)。 |
| **Backup Plan** | **備案(BCP)** | **(必填)** 萬一這工具徹底掛了，業務單位如何**人工**維持運作？ |
| **Handover** | 交接清單 | 程式碼在哪 (Git)？Config 檔在哪？排程在哪台機器？ |

### 💡 實戰調整建議 (Next Step)

這份架構把「商業獲利」拿掉，換成了**「責任保護」**。

* **如果你的專案是 "S" (個人工具)**：
重點寫 **A (Scope Out)** 和 **F (Troubleshooting)**。
* *理由：* 寫清楚「我不做什麼」避免同事一直加需求；寫清楚「怎麼修」方便你未來（或接手的人）忘記時查閱。


* **如果你的專案是 "M" (部門工具)**：
重點加強 **B (Access Control)** 和 **D (Input Format)**。
* *理由：* 多人使用最怕「亂改資料」和「匯入格式錯誤」，這兩點定義好，可以減少 80% 的維護工作。
